Gom Gmail · Outlook · Google Calendar · Outlook Calendar · Google Drive · OneDrive · Jira và Telegram về 1 nơi. 1. Vấn đề & Giải pháp

Vấn đề hiện tại

  • Mỗi ngày user phải mở 6-7 app: Gmail (cá nhân), Outlook (công ty), 2 Calendar, Drive, OneDrive, Jira...
  • Thông tin về cùng 1 dự án bị tản mác qua nhiều hệ sinh thái Google + Microsoft + Atlassian.
  • Không có cách gom nhóm theo context (dự án / khách hàng), chỉ gom theo nguồn.
  • Khi đi đường, mở điện thoại muốn ghi nhanh task → phải vào app, không tiện.
  • Trong team, mỗi người gom riêng — không chia sẻ được "view tổng thể" của 1 dự án cho đồng nghiệp. Muốn show cho sếp thấy mình đang theo dõi gì cũng phải chụp màn hình.
  • Khi triển khai cho nhiều user, không có ai quản trị tổng (xem stats, khoá account spam, debug khi user kêu sync hỏng).

Giải pháp Workspace Hub

  • 1 app kết nối 3 hệ sinh thái: Google (Mail/Calendar/Drive), Microsoft (Outlook Mail/Calendar/OneDrive), Atlassian (Jira).
  • Mỗi email / event / file / ticket / note = 1 Item đồng nhất.
  • User tạo Folder theo context: "Dự án A", "Khách B", "Đồ án TN".
  • Kéo thả Item đa nguồn vào Folder — Kanban view "Cần xem / Đang xử lý / Done".
  • Telegram bot 2 chiều: gửi notification cho user khi có email quan trọng, nhận message từ user để tạo Item nhanh.
  • User bật/tắt từng service riêng (không cần dùng hết) qua Connection Toggle.
  • Folder Share (Viewer-only): chủ folder mời teammate xem — họ thấy metadata Item (title, source, status, due) nhưng không xem được body email gốc (privacy).
  • Admin role: 1 user role Admin riêng có dashboard quản trị — xem stats sync per user, khoá/mở account, force-disconnect integration khi user kêu hỏng, xem log Telegram webhook.

2. Concept cốt lõi

3 object đồng nhất hóa thông tin đa nguồn + 1 object quản lý quyền. Đây là phần làm app có giá trị độc lập, không phải proxy thuần.

ITEM

Item

Card đại diện cho 1 đơn vị thông tin. 5 type:

  • Email — từ Gmail / Outlook
  • Event — từ Google / Outlook Calendar
  • File — từ Drive / OneDrive (Doc, Sheet, PDF...)
  • Ticket — từ Jira
  • Note — tạo trong app hoặc qua Telegram

Trường: type, source, source_id, mime_type (cho file), web_url, status, is_important.

FOLDER

Folder

Container chứa Items theo context. Đặt tên theo:

  • Dự án ("Dự án A")
  • Khách hàng ("Khách B")
  • Chủ đề ("Họp tuần")

1 Item có thể nằm nhiều Folder (m-n). Folder có name, color, icon. Kanban 3 cột bên trong.

TAG

Tag

Label nhẹ gắn xuyên Folder để filter:

  • #urgent
  • #cần-trả-lời
  • #tuần-này

Khác Folder: Tag là filter chéo. 1 Item nhiều tag được.

+ Phân quyền (Authorization)

2 lớp phân quyền độc lập: RBAC ở cấp hệ thống (ai được vào trang nào) và Resource-based ở cấp Folder (ai làm gì với 1 folder cụ thể).

ROLE

System Role (RBAC)

Mỗi user có ít nhất 1 role hệ thống. Seed 2 role mặc định:

  • User — role mặc định khi đăng ký. Đầy đủ tính năng workspace (Item/Folder/Tag/Compose/Telegram).
  • Admin — thêm quyền vào /admin: list user, khoá/unlock account, xem stats sync per user (số Item, last_synced), force-disconnect integration khi user báo hỏng, xem log Telegram webhook.

Quan trọng: Admin không xem nội dung email/file của user. Body lấy live qua Nango bằng token của Owner — Admin không có token đó.

Tech: ASP.NET Core [Authorize(Roles = "Admin")] + claim role trong JWT.

SHARE

Folder Share (Resource-based)

Quyền gắn vào 1 resource (Folder cụ thể), không phải toàn hệ thống. 3 vai trò trên 1 folder:

  • Owner — người tạo. Đổi tên, xoá, mời/gỡ người, đổi quyền.
  • Viewer scope đồ án — chỉ xem metadata Item (title, source, due_date, status). Không drag, không kéo Kanban, không xem body email.
  • Editor — đã loại ra khỏi scope đồ án (xem §10 Risks) để giảm complexity.

Viewer chỉ thấy metadata: vì body email lấy live qua Nango bằng Google/Outlook token của Owner — share token sẽ leak privacy. Metadata thì đã lưu sẵn trong DB nên share an toàn.

Tech: IAuthorizationHandler + FolderPermissionRequirement, kiểm tra ở mọi API có folder_id.

4. Tính năng chi tiết

01

Workspace Hub CORE

Item + Folder + Drag/drop. Sync data từ 7 nguồn về Item đồng nhất. User tạo Folder, kéo thả Item vào. Mỗi Folder có Kanban 3 cột.

  • Inbox view: list Item từ mọi nguồn
  • Folder view: Kanban "Cần xem / Đang xử lý / Done"
  • 1 Item có thể vào nhiều Folder
  • Drag/drop đổi Folder + đổi status
  • Tạo Note nội bộ (không có ở Gmail/Outlook)
React + @dnd-kit · EF Core m-n · Sync polling 5 phút · 5 Item type
02

Multi-Provider Hub CORE

Kết nối 3 OAuth provider (Google, Microsoft, Atlassian) qua Nango Cloud. Mỗi provider expose nhiều service, user bật/tắt từng cái độc lập qua Connection Toggle.

  • Google: Gmail · Calendar · Drive (bao Doc/Sheet/Slides)
  • Microsoft: Outlook Mail · Outlook Calendar · OneDrive (bao SharePoint docs)
  • Atlassian: Jira Cloud (tickets)
  • Settings page: list service + toggle on/off, hiển thị last-synced
Bảng ServiceConnection · ISyncProvider interface · 7 wrapper class lặp pattern
03

Schedule Send (multi-provider)

User viết email bây giờ, set giờ gửi sau. Hệ thống tự gửi đúng thời điểm — qua Gmail hoặc Outlook.

Phương thức:

  1. FE compose mail → chọn account gửi (Gmail/Outlook) → set send_at
  2. BE tạo record ScheduledEmail với status=pending
  3. External cron (cron-job.org) gọi POST /api/internal/process-scheduled mỗi 5 phút
  4. BE query WHERE send_at <= NOW() AND status=pending
  5. Tùy provider → gọi Gmail API hoặc Microsoft Graph qua Nango → update status=sent hoặc failed + log
Tại sao external cron, không phải IHostedService?
Free tier (Render free) sleep service sau 15p không request → background polling sẽ dừng → mail bị miss. External cron đánh thức service mỗi 5p, vừa giữ ấm, vừa chạy job. Không cần Hangfire/Quartz.
04

Telegram 2 chiều

Bot vừa gửi notification cho user, vừa nhận message từ user để tạo Item.

Chiều OUT — Bot gửi noti khi:

  • Có email từ Important Contact đến
  • Có Jira ticket mới được assign
  • Schedule Send hoàn tất

Chiều IN — User text bot → tạo Item:

  1. User text vào bot mình tự tạo: /task Viết báo cáo chương 3
  2. Telegram POST update tới webhook BE: POST /api/webhook/telegram
  3. BE verify chat_id, parse message → tạo Item type=note với title từ text
  4. Item xuất hiện trong Inbox app (2 giây sau)

User tự tạo bot — hướng dẫn trong app:

  1. User mở Telegram → search @BotFather → chat /newbot
  2. Đặt tên bot → nhận BOT_TOKEN
  3. Vào app Settings → Telegram → paste BOT_TOKEN → bấm "Generate link code"
  4. App tạo 1 mã link_token (vd LNK-A8K2P)
  5. User mở bot mình vừa tạo, chat /start LNK-A8K2P
  6. Telegram POST đến webhook BE → BE map chat_id ↔ user_id → lưu DB
  7. Done — từ đó user text bất kỳ lệnh nào, app biết là của họ
Vì sao mỗi user 1 bot riêng: Quyền riêng tư + tránh quota chung. User tự kiểm soát bot của mình, app chỉ lưu token đã mã hóa (AES) trong DB.
05

Important Contacts

User đánh dấu danh sách email + Jira reporter quan trọng (sếp, khách hàng, PM). Khi có email/ticket từ họ → badge đỏ trong app + Telegram bot ping.

  • CRUD Important Contact (email hoặc Jira account_id, label)
  • Match khi sync mail (case-insensitive, partial domain) + sync Jira
  • Tự tạo Notification + gửi Telegram khi match
06

Custom Tags

Tag riêng của app (không phải Gmail label / Jira label). Gắn vào mọi Item (email/event/file/ticket/note), xuyên Folder.

  • CRUD Tag (name, color)
  • Gắn nhiều tag vào 1 Item
  • Filter Item theo tag ở bất kỳ view nào
07

Phân quyền & Chia sẻ CORE

2 lớp authorization độc lập: RBAC hệ thống (Admin/User) + Resource-based trên Folder (Owner/Viewer).

RBAC — System Role:

  • Seed 2 role: Admin, User (default khi register)
  • Admin có trang /admin: list user, lock/unlock, force-disconnect integration, xem stats sync per user, log Telegram webhook
  • Admin không xem nội dung email/file — body cần Nango token của Owner
  • JWT chứa claim role, middleware [Authorize(Roles="Admin")] chặn ở Controller

Folder Share — Resource-based:

  1. Owner mở Folder → click Share → nhập email user khác (phải đã có account)
  2. BE tạo record FolderShare với permission='Viewer', accepted_at=null
  3. Notification in-app + (optional) email mời gửi đến invitee
  4. Invitee chấp nhận → folder xuất hiện trong sidebar tab Shared with me với badge "Read-only"
  5. Mọi API động vào folder/Item của folder đi qua FolderAuthorizationHandler: Owner full quyền, Viewer chỉ GET (LIST + READ metadata)
Tại sao Viewer-only, không có Editor? Editor cần handle race condition (2 user cùng kéo Kanban), phải lock optimistic + xử lý conflict — tốn 1 tuần riêng. Viewer-only đủ chứng minh resource-based authorization pattern cho rubric, và quan trọng hơn là privacy-safe: Viewer chỉ thấy metadata, không touch live data của Owner.
Role + UserRole + FolderShare · Policy-based Authorization · IAuthorizationHandler + FolderPermissionRequirement · seed Admin qua migration · FE: Admin page + Share modal + "Shared with me" section
Đã cân nhắc và BỎ — Email Templates + Folder Editor role. Templates: value thấp, user vào app không phải để gửi mail chính. Editor role: tốn ~1 tuần xử lý race condition khi 2 user cùng drag Kanban — Viewer-only đã đủ pattern cho rubric. Tập trung 7 tính năng trên là vừa scope.

12.1. Tech Backend

Nango INTEGRATION

Là gì: dịch vụ SaaS giúp app kết nối Gmail / Calendar / Slack / Notion / Jira / ... mà không cần code OAuth flow phức tạp.

Cách dùng trong project: đăng ký tài khoản free trên nango.dev → cấu hình Google + Microsoft + Atlassian provider 1 lần → FE gọi nango.auth('google-mail') → popup hiện ra cho user authorize → BE nhận về connection_id → gọi nango.proxy({ endpoint: '/gmail/v1/...' }) để hit Gmail API.

Vì sao chọn: OAuth 2.0 viết tay rất khó (refresh token, scope, error handling). Với 3 provider phải làm 3 lần — Nango làm sẵn cả 3. Free tier dư.

Microsoft Graph API

Là gì: API thống nhất của Microsoft cho mọi service: Outlook Mail, Calendar, OneDrive, SharePoint, Teams, Excel... 1 endpoint base https://graph.microsoft.com/v1.0/... cho tất cả.

Vd: GET /me/messages (Outlook mail) · GET /me/events (calendar) · GET /me/drive/items (OneDrive) · GET /sites/{id}/drive (SharePoint docs).

Vì sao quan trọng: 4 service Microsoft (Outlook/Calendar/OneDrive/SharePoint) chỉ cần 1 OAuth, 1 base URL, 1 schema. Đỡ rất nhiều so với từng app riêng.

Azure AD App Registration OAUTH SETUP

Là gì: chỗ đăng ký app của mình với Microsoft để dùng được Microsoft Graph. Vào portal.azure.com → Azure Active Directory → App registrations → New registration.

Cần config: redirect URI (tự Nango cung cấp), API permissions (chọn Mail.Read, Calendars.Read, Files.Read, Sites.Read.All), Authentication mode (Public client).

Free vĩnh viễn, không cần subscription Azure thật. Setup khoảng 30-45 phút lần đầu. Vướng nhiều ở consent screen — nên tham khảo doc Nango có sẵn.

Atlassian Developer Console OAUTH SETUP

Là gì: nơi đăng ký app với Atlassian để dùng được Jira/Confluence API. Vào developer.atlassian.com → My apps → Create → OAuth 2.0 (3LO).

Cần config: Authorization callback URL, scope (chọn read:jira-work, write:jira-work, read:jira-user), distribution mode (Development).

Free. Setup ~30 phút. Lưu ý: app ở mode "Development" có giới hạn 1 instance Jira, đủ cho project sinh viên.

Webhook CALLBACK

Là gì: ngược chiều với API call thông thường. Thay vì app gọi 3rd-party API, 3rd-party gọi vào endpoint của app khi có sự kiện. Vd: user text vào Telegram bot → Telegram POST JSON sang webhook BE.

Setup Telegram webhook: 1 lần gọi POST https://api.telegram.org/bot<TOKEN>/setWebhook với body { url: 'https://your-be.com/api/webhook/telegram', secret_token: '...' }

Yêu cầu: webhook URL phải public + HTTPS. Render free đáp ứng được. Khi dev local, dùng ngrok để expose localhost ra public.

FluentValidation VALIDATION

Là gì: thư viện validate object .NET kiểu "fluent" (chain method). Thay decorate [Required] trên DTO, viết Validator class riêng:

RuleFor(x => x.Email).NotEmpty().EmailAddress();

Vì sao dùng: rule phức tạp dễ viết hơn DataAnnotation. Test dễ. Tách validation ra khỏi DTO.

AutoMapper MAPPING

Là gì: map giữa Entity (DB model) ↔ DTO (API model) tự động.

Khai báo 1 lần: CreateMap<Item, ItemDto>() — sau đó mapper.Map<ItemDto>(item) tự copy field cùng tên.

Đỡ phải viết dto.X = entity.X; hàng chục lần. Khi DTO/Entity nhiều field, tiết kiệm hàng trăm dòng.

Serilog LOGGING

Là gì: thư viện log có cấu trúc cho .NET. Thay Console.WriteLine bằng:

Log.Information("User {UserId} synced {Count} items from {Source}", userId, count, source);

Output JSON, search được theo field.

Khi deploy production, mở log thấy ngay vấn đề. Filter theo level (Info/Warning/Error), ghi nhiều nơi cùng lúc (file + console + Seq).

RBAC vs ABAC AUTH

RBAC (Role-Based Access Control): quyền gắn vào role, user gán role nào thì có quyền của role đó. Vd: role Admin → vào được /admin. Đơn giản, dễ implement.

ABAC (Attribute-Based Access Control): quyền tính theo attribute của user + resource + context (vd user.department == resource.department). Linh hoạt nhưng phức tạp.

Project mình dùng RBAC ở cấp hệ thống (Admin/User) và resource-based ở cấp Folder (kiểu lai giữa RBAC và ABAC: permission tính theo user × folder cụ thể). Đủ cho rubric, không over-engineer.

Policy-based Authorization (ASP.NET Core) AUTH

Là gì: cơ chế ủy quyền của ASP.NET Core. Thay vì hard-code if (user.role != "Admin") return 403 trong từng controller, define policy 1 lần ở Program.cs:

options.AddPolicy("AdminOnly", p => p.RequireRole("Admin"));

Rồi gắn lên endpoint: [Authorize(Policy="AdminOnly")].

Logic auth tập trung 1 chỗ → dễ audit, dễ thay đổi (chỉ cần edit policy, không phải search/replace khắp codebase). Khi defense giảng viên hỏi "giả sử mai chuyển thành 3 role thì sao?", trả lời được: chỉ thêm policy mới.

IAuthorizationHandler AUTH

Là gì: interface trong ASP.NET Core để viết logic auth phức tạp (resource-based, condition-based). Mỗi handler:

  1. Define 1 IAuthorizationRequirement (vd FolderPermissionRequirement(string action))
  2. Define AuthorizationHandler<TRequirement, TResource>: nhận user + folder, check DB, gọi context.Succeed() hoặc không
  3. Đăng ký ở DI

Trong controller: await _auth.AuthorizeAsync(User, folder, "CanReadFolder");

Pattern chuẩn cho resource-based auth — cần biết user nào với folder nào. Khác policy-based đơn thuần (chỉ biết user). Defense slide tốt: trước/sau khi có handler, code controller sạch hơn nhiều (không còn if auth rải rác).

JWT Claims AUTH

Là gì: các trường thông tin nhúng trong JWT payload. Mỗi claim là 1 cặp key-value. Vd:

{ "sub": "user_123", "email": "a@b.com", "role": "Admin", "exp": 1735689600 }

ASP.NET Core đọc claim từ JWT vào User.Claims. RequireRole("Admin") nội bộ là check User.HasClaim("role", "Admin").

Trong project: claim role sync với DB lúc login. Lock user → middleware riêng vẫn check DB mỗi request (vì JWT không revoke được tức thì, TTL 1h).

12.2. Tech Frontend

Vite BUILD TOOL

const { data, isLoading } = useQuery({ queryKey: ['items'], queryFn: fetchItems });

Giảm 50% code FE. Cache 5p tự động, tự refetch khi user focus tab. Khi mutation (POST/PATCH) thành công, tự invalidate cache.

@dnd-kit/core DRAG/DROP

Là gì: thư viện drag/drop modern cho React. Wrap component bằng hook useDraggable / useDroppable.

Vì sao chọn: react-beautiful-dnd deprecated 2024 (Atlassian dừng maintain). dnd-kit nhẹ, support touch (mobile), accessibility tốt — chuẩn 2025-2026.

Tailwind CSS STYLING

Là gì: CSS framework utility-first. Thay file CSS riêng, viết class trong JSX:

<button className="px-4 py-2 bg-blue-500 rounded">

Không switch giữa CSS và JSX, không phải đặt tên class, responsive dễ (md:px-6), bundle nhỏ.

Là gì: bộ component đẹp sẵn, dùng với Tailwind. Khác MUI/AntD: chạy 1 lệnh npx shadcn-ui add button → copy code component vào project (không phải install package).

Customize được hết (code nằm trong repo mình), không lock vào API thư viện. Stack chuẩn 2025-2026.

react-hook-form + Zod FORM

react-hook-form: quản lý form không phải useState cho từng input. register('email') để bind.

Zod: define schema validation, tự infer TypeScript type: z.object({ email: z.string().email() }).

Kết hợp qua zodResolver → form vừa validation, vừa type-safe.
